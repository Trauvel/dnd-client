# Формат JSON сценария для генерации нейросетью

Документ описывает структуру JSON сценария настольной ролевой игры (книга мастера): локации, ситуации, переходы между ними. Формат предназначен для генерации или правки сценария нейросетью и последующего импорта в приложение.

---

## 1. Корневая структура

```json
{
  "title": "Название сценария",
  "description": "Краткое описание сюжета, сеттинга, уровня персонажей (опционально).",
  "scriptData": { ... },
  "npcs": [ ... ],
  "exportedAt": "2025-03-12T12:00:00.000Z"
}
```

Для генерации/правки сценария обязательны только **title** и **scriptData**. Остальные поля опциональны.

- **scriptData** — ядро сценария: локации, ситуации и переходы (ветки). В экспорте/импорте **не включаются** ссылки на изображения и аудио (audioIds, mapFileIds у локаций) — медиа добавляются и удаляются только вручную в редакторе сценария.
- **npcs** — краткие описания NPC (имя, тип, роль, описание); при импорте используются как подсказки, сами NPC создаются пользователем в интерфейсе и привязываются к локациям по id.
- **exportedAt** — дата экспорта в ISO; при импорте не обязательно.

---

## 2. scriptData — локации, ситуации, переходы

```ts
scriptData: {
  locations: Location[];
  situations: Situation[];
  branches: Branch[];
  startLocationId: string | null;
  encounters?: Encounter[];   // структурированные встречи (бои)
  checks?: Check[];            // структурированные проверки навыков
}
```

### 2.1 Локация (Location)

Локация — это место в мире сценария (таверна, подземелье, площадь).

| Поле       | Тип     | Обязательное | Описание |
|-----------|---------|---------------|----------|
| id        | string  | да            | Уникальный идентификатор (латиница, цифры, подчёркивание). |
| title     | string  | да            | Название локации. |
| body      | string  | да            | Текст сценария: что видит партия, что говорить мастеру, что могут сделать игроки. |
| notes     | string  | нет           | Заметки мастера (только для мастера). |
| order     | number  | нет           | Порядок отображения (целое число). |
| npcIds    | string[]| нет           | Список id NPC, которые относятся к этой локации (при импорте id могут не совпадать с реальными NPC — привязку делают в интерфейсе). |
| audioIds  | string[]| нет           | Id аудиофайлов локации. В экспорт/импорт **не входят** — задаются только в редакторе. |
| mapFileIds| string[]| нет           | Id вложений (карты/изображения) локации. В экспорт/импорт **не входят** — задаются только в редакторе. |

Пример:

```json
{
  "id": "loc_tavern",
  "title": "Таверна «Драконья пасть»",
  "body": "Низкий зал, камин, несколько столиков. Хозяин за стойкой. Игроки могут спросить о дороге, нанять проводника или подраться.",
  "notes": "Проводник — сюжетный квест в лес.",
  "order": 1,
  "npcIds": []
}
```

### 2.2 Ситуация (Situation)

Ситуация — ветка сценария: «если игроки сделали X» или отдельный эпизод. Всегда привязана к одной локации через **locationId** (логически «внутри» этой локации), но переходы (branches) могут вести в другие локации или ситуации.

| Поле        | Тип     | Обязательное | Описание |
|-------------|---------|--------------|----------|
| id          | string  | да           | Уникальный идентификатор. |
| title       | string  | да           | Короткое название (подпись на кнопке выбора в игре). |
| body        | string  | да           | Текст: что происходит, что говорить, варианты. |
| locationId  | string  | да           | Id локации, к которой относится ситуация. |
| order       | number  | нет          | Порядок внутри локации. |
| encounterId | string  | нет          | Id встречи (боя) из scriptData.encounters — при выборе этой ситуации мастер может запустить бой по шаблону. |
| checkIds    | string[]| нет          | Id проверок навыков из scriptData.checks — отображаются в книге мастера (навык, DC, успех/провал). |
| tags        | string[]| нет          | Теги сцены для фильтрации и поиска: `["combat", "ambush"]`, `["social", "quest"]`. Помогает нейросети балансировать сценарий и искать типы сцен. |

Пример:

```json
{
  "id": "sit_tavern_fight",
  "title": "Драка в таверне",
  "body": "Один из посетителей задевает персонажа. Можно принять вызов, откупиться или уйти. При драке хозяин вызывает стражу.",
  "locationId": "loc_tavern",
  "order": 1,
  "tags": ["combat", "ambush"]
}
```

### 2.3 Переход (Branch)

Переход задаёт связь «откуда → куда»: из локации в локацию, из локации в ситуацию, из ситуации в ситуацию или из ситуации в локацию.

| Поле    | Тип   | Обязательное | Описание |
|---------|-------|---------------|----------|
| id      | string| да            | Уникальный идентификатор перехода. |
| fromType| string| да            | Тип источника: **"location"** или **"situation"**. |
| fromId  | string| да            | Id локации или ситуации-источника. |
| toType  | string| да            | Тип цели: **"location"** или **"situation"**. |
| toId    | string| да            | Id локации или ситуации-цели. |
| label   | string| нет           | Подпись перехода (например текст кнопки). |

Правила:

- **fromId** и **toId** должны существовать: для fromType `"location"` — id из `locations`, для `"situation"` — id из `situations`; то же для toType/toId.
- Типичные связи:
  - локация → локация: переход в другое место (например «Выйти на площадь»).
  - локация → ситуация: выбор ветки внутри локации («Поговорить с хозяином», «Подраться»).
  - ситуация → ситуация: развитие ветки («Принять вызов» → «Бой»).
  - ситуация → локация: после ситуации партия переходит в новое место («После боя — в камере»).

Примеры:

```json
{ "id": "br_1", "fromType": "location", "fromId": "loc_tavern", "toType": "situation", "toId": "sit_tavern_fight", "label": "Ввязаться в драку" },
{ "id": "br_2", "fromType": "situation", "fromId": "sit_tavern_fight", "toType": "location", "toId": "loc_jail", "label": "Стража уводит в камеру" }
```

### 2.4 Стартовая локация

В **scriptData.startLocationId** указывается id локации, с которой мастер начинает игру. Должна быть в списке **locations**. Если не указана или пустая строка — старт не задан.

### 2.5 Встреча (Encounter)

Структурированное описание встречи (чаще всего бой). Позволяет нейросети генерировать сбалансированные столкновения, а в игре — подставлять участников боя по шаблону.

| Поле     | Тип    | Обязательное | Описание |
|----------|--------|--------------|----------|
| id       | string | да           | Уникальный идентификатор (префикс `enc_`). |
| type     | string | нет          | Тип встречи, например `"combat"`. |
| enemies  | array  | да           | Массив врагов: `{ "type": "bandit", "count": 4 }` (type — название типа из бестиария или сценария; count — количество). |
| xp       | number | нет          | Опыт за победу (суммарный или за встречу). |

Пример:

```json
{
  "id": "enc_bandits_tavern",
  "type": "combat",
  "enemies": [
    { "type": "bandit", "count": 4 }
  ],
  "xp": 200
}
```

Ситуация привязывает встречу через **encounterId** (id из scriptData.encounters).

### 2.6 Проверка (Check)

Структурированная проверка навыка (восприятие, скрытность и т.д.). В книге мастера отображается как блок «Проверка [навык] DC N. Успех: … Неудача: …»; позже возможен бросок из интерфейса.

| Поле    | Тип   | Обязательное | Описание |
|---------|-------|--------------|----------|
| id      | string| да           | Уникальный идентификатор (префикс `chk_`). |
| skill   | string| да           | Навык: perception, stealth, athletics, insight и т.д. (латиницей или как в правилах). |
| dc      | number| да           | Сложность проверки (целое число). |
| success | string| нет          | Текст при успешном броске. |
| fail    | string| нет          | Текст при провале. |

Пример:

```json
{
  "id": "chk_ambush",
  "skill": "perception",
  "dc": 13,
  "success": "Замечают засаду на дороге.",
  "fail": "Попадают в засаду."
}
```

Ситуация (или локация) привязывает проверки через **checkIds** — массив id из scriptData.checks.

## 3. Связи между сущностями (схема)

```
locations[]          situations[]
    │                      │
    │  locationId          │
    └──────────────┬───────┘
                   │
    branches[]: fromType/fromId → toType/toId
    - location → location  (переход в другое место)
    - location → situation (выбор ветки в локации)
    - situation → situation (развитие ветки)
    - situation → location (после ветки — новая локация)

startLocationId → id одной из locations
npcIds в location → привязка NPC к локации (id NPC в приложении задаются отдельно)

encounters[]          checks[]
    ↑                      ↑
    │ encounterId          │ checkIds[]
situation (или location)  situation (или location)
```

## 4. NPC в формате экспорта/импорта

Поле **npcs** в корне — массив **полных данных NPC** (все поля кроме аватарки). При экспорте из приложения выгружаются все поля; при импорте по каждому элементу создаётся карточка NPC, а **scriptData.locations[].npcIds** автоматически подменяются на новые id.

- **id** — обязателен для маппинга: локации ссылаются на NPC по id; при импорте создаются новые NPC с новыми id, ссылки в локациях обновляются.
- Аватар (изображение) в экспорт/импорт **не входит** — задаётся вручную в редакторе.
- **npcKinds**: массив **"enemy"** | **"ally"** | **"story"**.

Пример (полный набор полей):

```json
"npcs": [
  {
    "id": "npc_tavern_keeper",
    "name": "Хозяин таверны",
    "type": "Средний гуманоид, нейтральный",
    "npcKinds": ["ally"],
    "armorClass": 11,
    "hpText": "9 (2d8)",
    "speed": "30 ft.",
    "description": "Знает дорогу в лес, может дать квест.",
    "notes": "Сюжетный проводник."
  }
]
```

Можно указывать любые поля карточки NPC: strength, dexterity, constitution, intelligence, wisdom, charisma, skills, senses, languages, xp, challenge, habitat, traits, abilities, actions, legendaryActions и т.д.

## 6. Рекомендации для нейросети

1. **id** — уникальные в рамках всего сценария. Удобные префиксы: `loc_`, `sit_`, `br_`, `enc_`, `chk_`.
2. **locationId** у каждой ситуации должен указывать на существующую локацию.
3. В **branches** все **fromId** и **toId** должны ссылаться на существующие id в **locations** или **situations** в зависимости от fromType/toType.
4. **startLocationId** — один из id из **locations** или null.
5. **encounterId** в ситуации должен ссылаться на id из **scriptData.encounters** (если указан). **checkIds** — только существующие id из **scriptData.checks**.
6. Тексты **body** и **title** лучше задавать на языке сценария (например русский); **notes** — только для мастера.
7. **tags** у ситуации — произвольные строки (латиница или транслит): `combat`, `ambush`, `social`, `quest`, `trap`, `puzzle`. Нужны для фильтрации сцен в редакторе, баланса (сколько боёв/социалок) и поиска по типу сцены.
8. При правке существующего JSON сохраняйте структуру и id, если не требуется переименование сущностей; при добавлении новых сущностей добавляйте новые уникальные id и соответствующие ветки в **branches**.

После генерации или правки пользователь загружает JSON через «Импорт JSON» в приложении: можно создать новый сценарий или заменить книгу мастера (локации, ситуации, переходы) в выбранном сценарии.
