# Визуальный редактор сценария — спецификация

## 1. Обзор

Редактор строится поверх текущей модели сценария (`ScenarioScriptData`: locations, situations, branches, startLocationId). Добавляется слой **расположения на полотне** и расширения для карт и связей NPC — всё в JSON для экспорта/импорта и работы с нейросетью.

---

## 2. Полотно и блоки

- **Полотно (canvas)** — бесконечная или большая область с панорамированием (pan) и зумом. Блоки можно перетаскивать.
- **Блоки локаций** — прямоугольные карточки на полотне. Внутри блока — **ячейки ситуаций** (список или сетка). Ситуации визуально привязаны к своей локации.
- **Линии переходов** — рёбра между узлами:
  - ситуация → ситуация (внутри локации или между локациями);
  - ситуация → локация (переход «в другую локацию»);
  - локация → локация (прямой переход между локациями).
  Источник истины — массив `branches` (fromType/fromId, toType/toId). Визуально линии рисуются от центра ячейки/блока к центру целевого узла.
- **Блоки NPC** — отдельные узлы на полотне. Связь NPC ↔ локация: линия от блока NPC к блоку локации (локации уже хранят `npcIds`). Связи NPC ↔ NPC — отдельный тип связи (см. раздел «Расширение данных»).

Для каждой сущности (локация, ситуация, NPC) — **контекстное меню** (ПКМ): редактировать, удалить, добавить переход (для локации/ситуации), привязать NPC к локации, открыть редактор карты (для локации) и т.д.

---

## 3. Расширение JSON (для layout и новых сущностей)

Текущий `scriptData` остаётся основой. Добавляется опциональный блок **view** (или хранится отдельно на клиенте и при экспорте объединяется в один JSON):

```ts
// Опциональный слой визуального редактора (можно хранить в scriptData.view)
interface ScenarioScriptView {
  /** Позиции и размеры блоков на полотне */
  nodes?: {
    location?: Record<string, { x: number; y: number; w?: number; h?: number }>;
    situation?: Record<string, { x: number; y: number }>; // относительно локации или абсолютно
    npc?: Record<string, { x: number; y: number; w?: number; h?: number }>;
  };
  /** Связи NPC–NPC (id связи, от кого, к кому, подпись) */
  npcRelations?: Array<{ id: string; fromNpcId: string; toNpcId: string; label?: string }>;
}
```

- Если `view` нет — редактор строит авто-раскладку по графу (например, локации в ряд/сетку, ситуации внутри блоков).
- Экспорт в JSON для нейросети: полный `scriptData` + `view` (опционально) + краткие NPC (уже есть в `ScenarioExportPayload`). Импорт: валидация и слияние с текущим сценарием.

**Encounters и Checks (для генерации нейросетью и игры):**

- **encounters[]** — структурированные встречи (бои). Каждая запись: id, type (например `"combat"`), enemies (массив `{ type: string, count: number }` или привязка к NPC сценария), xp. Ситуация или локация может ссылаться на встречу через `encounterId`.
- **checks[]** — структурированные проверки навыков. Каждая запись: id, skill (например `"perception"`), dc, success (текст при успехе), fail (текст при провале). Ситуация/локация ссылается через `checkIds[]` или `checkId`.
- В редакторе: в карточке ситуации (и при желании локации) — блок «Встреча» (выбор или создание encounter), блок «Проверки» (список checks с подписями DC/skill). На ноде ситуации — индикатор (иконка), если привязаны encounter или checks.
- **tags** у ситуации — массив строк (например `["combat", "ambush"]`) для фильтрации сцен, поиска по типу и баланса при генерации (сколько боёв/социалок/квестов). В редакторе: поле тегов в карточке ситуации, фильтр по тегам на полотне или в списке.
- В игре (после этапа «только книга»): по ситуации с encounterId — кнопка «Начать бой» с подстановкой участников; по checkId — отображение проверки и опционально кнопка «Бросить».

---

## 4. Локация: медиа

- **Изображения** — уже есть `mapFileIds` (вложения). Имеет смысл различать тип:
  - описательное изображение (атмосфера);
  - тактическая карта (для редактора карт и токенов).
  Предложение: в локации добавить опционально `mapFileId?: string` (одна выбранная тактическая карта для редактора карт) или пометку в `mapFileIds` по типу. Либо отдельный массив `tacticalMapFileIds` vs `imageFileIds`.
- **Аудио** — уже есть `audioIds[]`, несколько треков на локацию. В контекстном меню: «Добавить аудио» → выбор из загруженных.

---

## 5. NPC

- На полотне NPC — отдельный блок (карточка с именем, опционально аватар). Связь с локацией: линия к блоку локации; привязка через `loc.npcIds` (в контекстном меню локации — «Привязать NPC», в меню NPC — «Привязать к локации»).
- **Связи NPC–NPC**: например «союзник», «враг», «знает». Хранить в `scriptData.view.npcRelations` или в отдельном массиве `npcRelations` в `scriptData`. Визуально — линии между блоками NPC с подписью.

---

## 6. Редактор карт (отдельный инструмент)

- Открывается из контекстного меню локации: «Редактор карты» по выбранному изображению (тактическая карта).
- **Разметка** — слой поверх изображения. Варианты:
  - **Сетка ячеек**: разбить карту на клетки (например, квадраты или гексы), каждая ячейка — DOM-элемент (div) или SVG rect. У каждой ячейки: описание, заметки, при необходимости ссылка на ситуацию/NPC.
  - **Полигоны/зоны**: произвольные области (polygon/rect/circle), у каждой — описание и т.д.
- **Взаимодействие**: клик по ячейке/зоне — показать/редактировать описание (тултип, боковая панель или модалка). Данные разметки хранить в JSON (привязка к `locationId` и `mapFileId`):

```ts
interface MapMarkup {
  locationId: string;
  mapFileId: string;
  /** Сетка: ширина/высота в ячейках, размер ячейки в px или % */
  grid?: { cols: number; rows: number; cellSizePx?: number };
  cells?: Array<{ row: number; col: number; description?: string; situationId?: string }>;
  /** Или зоны (полигоны/прямоугольники) */
  zones?: Array<{ id: string; type: 'rect' | 'polygon' | 'circle'; points: number[]; description?: string }>;
}
```

В MVP достаточно сетки с ячейками и полем описания на ячейку. Полигоны — следующий этап.

---

## 7. Сценарий в JSON и нейросеть

- **Экспорт**: `ScenarioExportPayload` (title, description, scriptData без audioIds/mapFileIds у локаций, npcs). Изображения и аудио в экспорт не включаются — добавляются только вручную в редакторе. При наличии добавить в экспорт: `view`, `mapMarkup`.
- **Импорт**: загрузка JSON → валидация → сохранение script. Ссылки на медиа (audioIds, mapFileIds) при импорте не переносятся — после импорта карты и аудио привязывают вручную в редакторе.

## 8. MVP и дорожная карта

### Фаза 1 — Визуальный редактор сценария (MVP)

1. Полотно с pan/zoom, блоки локаций (позиция x,y в `view.nodes.location`).
2. Внутри блока локации — список ситуаций (ячейки); переходы (branches) отображаются линиями между ячейками/локациями.
3. Контекстное меню: локация (редактировать, удалить, задать старт, добавить ситуацию, привязать медиа/NPC, открыть редактор карты); ситуация (редактировать, удалить, добавить переход, привязать встречу/проверки); создание новой локации/ситуации с полотна.
4. Блоки NPC на полотне, связи NPC → локация (по npcIds). Контекстное меню NPC.
5. Сохранение/загрузка script + view через существующий API (расширить scriptData или отдельный endpoint для view).
6. **Encounters и Checks:** в scriptData — массивы `encounters[]`, `checks[]`; у ситуации (и при необходимости у локации) — поля `encounterId?`, `checkIds?`. В карточке ситуации: блок «Встреча» (выбор/создание encounter), блок «Проверки» (список checks). Импорт/экспорт JSON и док для нейросети — расширить под новую схему.

### Фаза 2 — Редактор карт

1. Открытие изображения локации в режиме «Редактор карты».
2. Сетка поверх карты (cols × rows), ячейки — интерактивные; у каждой ячейки описание (и при желании связь с ситуацией).
3. Сохранение разметки в JSON (mapMarkup по locationId + mapFileId).

### Фаза 3 — Тактика (Foundry-like 2D)

1. **Стены**: рисование отрезков на карте, сохранение в данных карты; в игре — блокировка движения и видимости.
2. **Объекты**: размещение токенов/декораций на карте (привязка к ячейке или координате).
3. **Туман войны**: ячейки помечены как открытые/закрытые для партии; при открытии — показывать содержимое.
4. **Угол обзора**: от персонажа (или от токена) расчёт видимой области (лучи/сегменты), отсечение стенами; показывать только видимое.

Технически: разметка стен и объектов хранится в той же структуре карты (например, `walls: Segment[]`, `objects: { id, x, y, type, ... }`). В игре — отдельный режим «тактическая карта» с движением токенов и расчётом видимости.

## 8.1 Интеграция encounters и checks в игру

- **Только книга мастера (первый этап):** для ситуации с encounterId показывать блок «Встреча: N врагов, X XP»; для checkIds — блок «Проверка [навык] DC N. Успех: … Неудача: …». Без автоматики.
- **В игре (второй этап):** по ситуации с encounterId — кнопка «Начать бой»: создание CombatState, подстановка участников (type → шаблон из бестиария или NPC сценария), токены. По проверке — опционально кнопка «Бросить» (бросок от имени партии/персонажа, показ успех/провал). Ветвление по результату проверки (success/fail → разные ситуации) — по желанию, позже.

## 9. Дополнительные идеи

- **Мини-карта**: на полотне визуального редактора — уменьшенная схема «все локации + связи» для быстрой навигации.
- **Поиск по сценарию**: поиск по тексту в локациях/ситуациях/NPC с переходом к узлу на полотне.
- **Валидация**: «сироты» (ситуации без локации, ветки на несуществующие id), предупреждение о дубликатах id при импорте.
- **Шаблоны сценария**: экспорт сценария как шаблона (без привязки к конкретным файлам); создание нового сценария «из шаблона» + загрузка своих медиа.
- **Версионирование**: при сохранении script сохранять историю версий (хотя бы последние N) для отката.
- **Роли/теги локаций**: тег «улица», «подземелье», «город» — для фильтров и подсказок нейросети при генерации.
- **Условия на переходах**: в ветке опционально `condition?: string` (текстовое описание или ключ условия) — «если игроки нашли ключ»; в игре можно использовать для скрытия вариантов до выполнения условия.

## 10. Стек и реализация

- **Полотно**: React + библиотека для графов/нод (React Flow, или своя отрисовка на SVG/Canvas). React Flow даёт ноды, рёбра, pan/zoom из коробки; кастомные ноды «локация с ячейками» и «NPC» делаются как отдельные компоненты.
- **Линии**: по данным `branches` строить рёбра; источник/цель — по fromId/toId и fromType/toType.
- **Состояние**: один источник истины — scriptData (+ view, mapMarkup); сохранение через `saveScenarioScript` (расширенный script) или отдельный endpoint для view/mapMarkup.
- **Контекстное меню**: единый компонент по типу сущности (location | situation | npc); пункты меню — действия (редактировать, удалить, добавить переход, привязать к локации и т.д.).

Если нужно, следующий шаг — схема API (расширение PUT script или новые эндпоинты для view/mapMarkup) и структура папок/компонентов в dnd-client.
